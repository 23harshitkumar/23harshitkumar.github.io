Welcome to my first GSoC 2026 blog post! This summer, I'm contributing to the **Accord Project**, focusing on the [`template-playground`](https://github.com/accordproject/template-playground) repository under the GSoC project "[Template Logic Support for Playground](https://summerofcode.withgoogle.com/programs/2026/projects/R4u8FyUV)". 

The first four weeks have been incredibly productive. My main goal has been to lay the foundational architecture for executing TypeScript logic for smart legal contracts in a sandbox environment directly inside the browser. Here is a week-by-week breakdown of the progress, highlighting the key Pull Requests (PRs) and Architectural Decisions that shaped the project.

## 🚀 Key Accomplishments (Weeks 1-4)

### Week 1: Transitioning to the Official Execution API (US-01)
My initial focus was building out the experimental Logic Editor interface. However, early on, my mentors and I made a crucial **Architectural Decision**: we needed to abandon the experimental, custom in-browser compilation strategy in favor of the official `@accordproject/template-engine` API.
- **The Shift**: By migrating to the official engine API, we guaranteed that logic written in the Playground would execute identically to logic running in production environments.
- **UI Hardening**: I began refactoring the UI, migrating away from legacy Tailwind CSS classes towards strict Ant Design components for a cleaner, more maintainable architecture. I also resolved architectural race conditions during Monaco editor initialization to ensure stable performance.

### Week 2: Refactoring Engine Compilation for Performance
Because the Playground compiles user logic on-the-fly, executing the full `template-engine` compilation step on every keystroke was far too heavy.
- **Architectural Decision (Hybrid Compilation Strategy)**: To fix this, I dove into the upstream `template-engine` repository. With guidance from my mentor, Sanket Shevkar, I decoupled the TS compilation step from the logic execution pipeline (init/trigger methods) and refactored the `compileLogic()`, `init()`, and `trigger()` methods to introduce an optional `useCache` parameter.
- **The Result**: This allowed legacy applications to continue using implicit compilation, while enabling the `template-playground` to manually trigger and cache compilations, resulting in a massive performance optimization and early error detection. 

### Week 3: Architecting the Sandboxed Execution Pipeline (US-03)
Executing user-provided TypeScript safely in the browser is dangerous. We needed an airtight sandbox.
- **Architectural Decision (Web Workers + iFrames)**: After days of heavy discussions with the mentors, we came up with a sandboxing pipeline using a hidden `iframe` coupled with a nested Web Worker (`logic-worker.ts`). This provided the much needed thread isolaion and preventing infiltration. By allowing `allow-scripts` and deliberately omitting `allow-same-origin`, the browser forces the iframe into a strict null origin, completely isolating it from the host application.
- **Security Hardening**: I enforced strict `Content-Security-Policy` headers (`default-src 'none'; script-src 'unsafe-inline' blob:`) on the sandbox to prevent data exfiltration.
- **Upstream Discoveries**: During this process, I discovered upstream validation gaps. I opened formal issues on the core `template-engine` (to enforce `TemplateLogic` subclassing) and the `concerto` repository (to fix TypeScript mapping for `DateTime` fields).

### Week 4: Adaptive UI Layouts & Runner Integration (US-08)
With the underlying sandbox execution pipeline finalized, the application needed an interface capable of interacting with the new engine.
- **Architectural Decision (Adaptive State)**: For **US-08**, I introduced a dynamic, responsive 3-column workspace. I integrated smart visibility state logic into the Zustand store. Now, when a user loads a logic-enabled template, the Playground automatically adapts—hiding the PDF preview and revealing the Logic Editor and Contract Runner Workspace.
- **Testing**: I maintained 100% component test coverage by writing comprehensive `vitest` unit tests for the new UI panels. 

## 🧠 Challenges Faced
The biggest challenge over these four weeks was navigating the intricate dependency tree of the Accord Project ecosystem. Changes in the `template-playground` frequently required digging deep into the `template-engine` and `concerto` core libraries. Understanding how `TemplateArchiveProcessor` dynamically evaluates logic and coming up with a robust and efficient sandboxing solution was time consuming and required a lot of deep-dive debugging and guidance from the mentors.

## 🔜 Next Steps
With the core sandboxed logic pipeline and the adaptive Logic Editor UI finalized, my next steps involve refining the interactive contract execution features (US-04). We will be building out the State and Event tracking panels so users can visually step through their contract's lifecycle.

A huge thanks to my mentors (Sanket Shevkar, Matt Roberts, and Priyanshu Singh) for their continuous support through these complex architectural decisions. Stay tuned for updates from upcoming weeks!
