Welcome to my first GSoC 2026 blog post! This summer, I'm contributing to the **Accord Project**, focusing on the [`template-playground`](https://github.com/accordproject/template-playground) repository under the GSoC project "[Template Logic Support for Playground](https://summerofcode.withgoogle.com/programs/2026/projects/R4u8FyUV)". 

The coding period for GSoC 2026 officially started on May 25. The first four weeks have been incredibly productive. My main goal has been to lay the foundational architecture for executing TypeScript logic for smart legal contracts, in a sandbox environment directly inside the browser. Here is a week-by-week breakdown of the progress, highlighting the key Pull Requests (PRs) and Architectural Decisions that shaped the project so far.


## 🚀 Key Accomplishments (Weeks 1-4)

### Week 1: Transitioning to the Official Execution API (US-01)

My initial focus was building out the experimental Logic Editor interface. However, early on, my mentors and I made a crucial **Architectural Decision**: we needed to abandon the experimental, custom in-browser compilation strategy in favor of the official `@accordproject/template-engine` API.

- **The Shift**: By migrating to the official engine API, we guaranteed that logic written in the Playground would execute identically to logic running in production environments.

- **UI Hardening**: I began refactoring the UI, migrating away from legacy Tailwind CSS classes towards strict Ant Design components for a cleaner, more maintainable architecture. Sanket also helped me in figuring out how can we utilize the sidebar navigation to switch between different parts of the application. 

Link to the first PR is [here](https://github.com/accordproject/template-playground/pull/923).

### Week 2: Implementing the Logic Compilation Pipeline (US-02)

This week was about integrating the logic editor with the template engine and build the compilation pipeline for the logic using the official class TemplateArchiveProcessor. But there was a problem.Because the Playground compiles logic on-the-fly, executing the full `template-engine` compilation step on every manual trigger was far too heavy. 

- **Architectural Decision (Hybrid Compilation Strategy)**: To fix this, I dove into the upstream `template-engine` repository. With guidance from my mentor, Sanket Shevkar, I decoupled the TS compilation step from the logic execution pipeline (init/trigger methods) and refactored the `compileLogic()`, `init()`, and `trigger()` methods to introduce an optional `useCache` parameter. Older apps will ignore it and behave exactly as they did before, but clients that want to opt into the optimization can just pass true to use the cached logic.

- **The Result**: This allowed legacy applications to continue using implicit compilation, while enabling the `template-playground` to manually trigger and cache compilations, resulting in a massive performance optimization and early error detection.

After Sanket cut the release for template-engine v3.1.0, I updated the dependencies of the template-playground and integrated the modified template-engine to use this hybrid compilation strategy. The PR for the same is [here](https://github.com/accordproject/template-playground/pull/928).

At this stage, both of the PRs were merged and now we could focus on the key aspect of this project i.e. The Sandboxing! 

### Week 3: Architecting the Sandboxed Execution Pipeline (US-03)

Executing user-provided TypeScript safely in the browser is dangerous. We needed an airtight sandbox.

- **Architectural Decision (Web Workers + iFrames)**: After days of heavy discussions with the mentors and research, we came up with a sandboxing pipeline using a hidden `iframe` coupled with a nested Web Worker (`logic-worker.ts`), which was spawned using blob URL, giving it the much needed `null` origin. This provided the much needed thread isolation and prevention of infiltration. By explicitly allowing `allow-scripts` and deliberately omitting `allow-same-origin`, the browser forces the iframe into a strict null origin, completely isolating it from the host application.

We got the PR merged which was a big achievement for the project. Link to the PR is [here](https://github.com/accordproject/template-playground/pull/932).

### Week 4: Adaptive UI Layouts & Runner Integration (US-08)

With the underlying sandbox execution pipeline finalized, the application needed an interface capable of interacting with the new engine i.e. The Execution Tools! 

As we entered Week 4 with several major PRs merged, we reassessed our timeline. The init/trigger execution functionalities (US-04/05/06) were still pending. Having already built and tested the core execution pipeline locally, I proposed a strategic pivot: we should prioritize building the final adaptive layout for the logic panels (US-08) before wiring up the execution methods. Laying down this UI foundation first made integrating the init/trigger logic (which were relatively straightforward changes), seamless. You can check out the PR for the Adaptive Layout [here](https://github.com/accordproject/template-playground/pull/933).

Currently, we are waiting for the final review and merge of this PR. As soon as it gets merged, I plan to push the changes for execution methods and be done with the Phase 1 of the project (core execution pipeline).


## Upstream Discoveries

During the development of the logic compilation and sandboxing pipelines (US-02 and US-03), I discovered a few validation gaps in the upstream Accord Project repositories. Instead of just building localized workarounds, I opened formal issues to improve the core ecosystem for everyone:

1. **`template-engine` Missing Subclass Validation**: The engine was silently compiling logic classes even if they failed to extend the required `TemplateLogic` base class. I opened an issue regarding the need to strictly enforce `TemplateLogic` subclassing. Catching this validation gap early ensures that developers write correctly structured logic classes *before* the engine attempts to compile them, preventing cryptic runtime errors down the line. Link to the issue is [here](https://github.com/accordproject/template-engine/issues/163).

2. **`concerto` TypeScript Mappings**: The code generator (concerto-codegen) currently maps `DateTime` fields strictly to `Date` objects in TypeScript. However, because our templates pass data as JSON (where dates are serialized as ISO strings), this strict typing causes false compiler errors. I opened an issue [here](https://github.com/accordproject/concerto/issues/1272) proposing it map to `string | Date` so the TypeScript definitions natively support real-world JSON execution.

Identifying and reporting these upstream discoveries has been one of the most rewarding parts of the project so far, improving the stability of the entire Accord Project stack!


## Challenges Faced

The most significant hurdle was managing the intense time pressure while navigating the complex architecture of the Accord Project ecosystem. Developing the template-playground wasn't happening in isolation; it required digging deep into the core template-engine and concerto libraries. Architecting a robust, secure sandboxing solution and understanding exactly how the TemplateArchiveProcessor dynamically evaluates logic required heavy deep-dive. Doing this level of complex, cross-repository engineering while the clock was ticking, was demanding. 

However, completing the heavy architectural foundations early ensures that we are perfectly positioned to accelerate feature delivery and be comfortably ahead of schedule.


## Next Steps

With the core sandboxed logic pipeline and the adaptive Logic Editor UI finalized, my next steps involve refining the interactive contract execution features and the UI/UX of the Playground.
